# Product System For 裁断

Use this reference when turning the diagnostic into a product, service, paid community, app, or content system.

## Product Definition

Do not position it as "AI business consultant."

Position it as:

> A反谄媚 commercial cross-examination system that separates itch from business, catches self-rationalization, and forces external validation.

Even more simply: the product is a decision brake before it is an advice engine.

Anti-positioning:

- Not a startup coach that gives encouragement.
- Not a prompt pack.
- Not a generic business plan generator.
- Not a strategy assistant that rushes to solutions.
- Not a "make money with AI" course.
- Not a community full of many agents as a feature buffet.

## Core User

Start with a narrow user:

- Solo creators, strategists, consultants, operators, or AI builders with many ideas but weak commitment.
- People smart enough to rationalize anything.
- People who say they want truth and can tolerate a painful but useful diagnosis.
- People with some ability to act within 7-14 days.

Avoid early users who only want motivation, emotional reassurance, passive learning, or vague inspiration.

## Product Layers

Build in this order:

1. **Public裁断内容**: Publish sharp teardown content to prove judgment and attract the right users.
2. **Manual concierge diagnosis**: Paid 1:1 or small-batch project裁断 using the intake and scorecard.
3. **裁断档案**: Store each user's assets, constraints, idea history, validation commitments when justified, stop/pause conditions, and repeated patterns.
4. **Validation loop**: Every diagnosis must create a decision state; only justified continuations create a task and return date.
5. **Case library**: Anonymized cases become reusable judgment assets.
6. **Community ritual**: Weekly live裁断, member提交, result回填, parking-lot reviews.
7. **Agent/app layer**: Only after the manual workflow shows repeat demand.

## MVP Workflow

Use a seven-step loop:

1. **Intake**: target user, payer, pain, offer, price, channel, assets, constraints, proof.
2. **Cross-exam**: ask the 3-5 questions that reveal the 当前卡口.
3. **Verdict**: red/yellow/green-ish with confidence, score, personal constraint, and stop/pause/narrow/proceed condition.
4. **Commitment check**: make the user decide whether they still want to continue after seeing the cost and risk.
5. **Wedge**: smallest testable paid offer or content proof, only if continuation is justified.
6. **Mission**: 7-14 day validation task with kill metric, only when the verdict is Yellow or Green-ish.
7. **Return**: result回填 and pattern update.

If any step is missing, the product becomes content, not a diagnostic system.

## User Value Loop

The product loop is:

1. Low-friction entry
2. 当前卡口
3. Portable裁断卡
4. Cautious choice: stop, pause, narrow, investigate, or continue
5. External mission, only when useful
6. Result回填
7. User pattern update
8. Sharper next裁断

If the loop stops at "impressive answer," the product will not retain.
If the loop jumps straight to "here is what to do," the product becomes ordinary AI advice.
If the loop creates a mission for every Red verdict, it is secretly encouragement.

## Memory Model

The moat is not "the prompt." The moat is accumulated user state plus cases.

Track:

- User assets
- User constraints
- Idea log
- Repeated self-rationalization patterns
- Validation commitments, if justified
- Results
- Avoidance signals
- Offers that produced payment
- Phrases users respond to

Every return visit should make the diagnosis more specific than a new chat.

## Content Funnel

Public content should demonstrate judgment, not teach broad frameworks.

Content formats:

- "这个副业为什么看起来对、做起来死"
- "这个 AI 产品哪里不是生意"
- "这个爆款案例真正的护城河是什么"
- "心痒和生意理由如何被混在一起"
- "我会如何用 14 天杀死/验证这个想法"

CTA should lead to a paid or near-paid diagnostic:

- "提交你的项目，接受一次裁断。"
- "如果你想被顺着说，不要来。"
- "如果你愿意拿 14 天换一个真实信号，来。"

## Monetization Ladder

Start simple:

1. Paid single裁断 report/session
2. 4-week validation sprint
3. Small high-trust paid group
4. Membership with weekly裁断 and result回填
5. App/agent after repeated workflow is proven

Do not start with a full app or a large community. That would turn product uncertainty into operations burden.

## Retention Logic

Users renew because the system remembers them and catches patterns sooner.

Retention assets:

- User档案
- Idea parking lot
- Weekly result回填
- Pattern callout
- Case library
- Member examples
- Evolving kill metrics

If the product only gives one impressive diagnosis, it is a service. If it repeatedly improves decisions over time, it can become a community or tool.

## Delivery Objects

Every paid interaction should create one of these. Do not force every interaction to create every object; choose the object that matches the verdict.

- 裁断卡
- 用户档案
- 验证任务（only for justified Yellow/Green-ish continuations）
- 停止/缩窄条件
- 回填表
- 停车场 list
- Pattern update
- Anonymous case note

These objects make the service feel tangible and allow the system to compound.

## First 30 Days

Recommended launch sequence:

- Week 1: Publish 3裁断 posts and offer 5 paid/discounted manual裁断 slots.
- Week 2: Deliver 5 manual裁断 cases and record contradictions, questions, and user language.
- Week 3: Require result回填 from those users; publish anonymized lessons.
- Week 4: Package a 4-week validation sprint for 5-10 users.

Kill or reframe if no one will pay for manual裁断 after seeing public examples.
